# 最小可验证变更（Roblox）

## 工作流
1. 先完成任务分类。
2. 选择验证级别并定义验证方法。
3. 写编码前行为声明。
4. 需要可测试时先做“红”（失败测试或重现）。
5. 只做最小“绿”实现。
6. 执行验证并给出结果。
7. 仅在必要时做受限重构。

## 1) 任务分类（先做）
在编码前明确类型：
- 逻辑 / 行为
- 错误修复
- 重构
- UI / 表现层
- 配置 / 内容
- 原型 / 试验

## 2) 选择验证级别

### 严格（测试优先，必须）
适用：
- 核心逻辑
- 状态机
- 伤害/计算系统
- 可稳定复现的 bug

规则：
- 先写一个失败测试或重现步骤
- 实现前必须确认失败且失败原因正确

### 轻度（推荐验证）
适用：
- 集成密集逻辑
- 事件驱动系统
- 客户端-服务端交互

规则：
- 优先最小重现/验证脚本
- 测试不自然时可不写测试，但必须有验证步骤

### 手动验证（可跳过严格测试）
适用：
- UI/视觉
- 内容/配置
- 一次性脚本

规则：
- 必须给出可执行手动检查清单

## 3) 编码前行为声明（必须）
开始编码前先写清楚：
- 添加/修复了什么行为
- 哪些行为不得改变
- 成功标准是什么
- 是否启用“严格测试优先”

## 4) 红（仅适用可测试任务）
- 写一个最小失败测试/重现
- 确认其因正确原因失败
- 在此之前不写生产代码

## 5) 绿（最小实现）
只写满足目标行为的最小代码。

硬规则：
- 不做推测性抽象
- 不加无需求选项/配置
- 不加重试/回退/额外守卫（除非已证明必需）
- 不扩散到无关范围

## 6) 验证（必须）
用已选方式直接证明行为有效：
- 单元测试
- 重现脚本
- 集成检查
- 手动清单

验证应针对行为结果，而非实现细节。

## 7) 重构（可选，且受限）
仅在以下条件都满足时允许：
- 行为已验证
- 变更局部且小
- 仅提升清晰度或去重

禁止：
- 大规模重组
- 跨文件重构
- 未请求的公共接口变更

## Roblox 项目适配约束

### 代码与工具约束
- 优先使用 `multi_edit` 修改脚本，避免无必要 `execute_luau` 直写。
- 使用 Roblox Luau 与最新 API。
- 保持手术式改动：只改需求相关代码，不顺手重构无关模块。
- 参数统一放在脚本或功能代码块头部，命名使用 Pascal 命名法。
- 注释用中文；为参数与功能块写简介，不为逐行实现写注释。

### 规则读取约束
执行任务前至少快速扫描与任务相关的规则文件：
- `rules/lifecycle.md`
- `rules/equipment-class.md`
- `rules/studio-workflow.md`
- `rules/known-modules.md`
- 若任务包含“审计/排查/review”：额外读 `rules/code-audit.md`
- 若任务包含“瘦身/精简/删冗余”：额外读 `rules/code-slimming.md`
- 做内存泄漏排查前：先读 `rules/stage8-clean-modules.md`，避免重复审计已证伪方向

### 验证优先级（Roblox）
- 行为/逻辑变更：优先提供最小重现或测试脚本。
- 事件链路或客户端-服务端问题：给出可执行回归步骤（进入 Play、触发条件、预期日志/状态）。
- UI/配置变更：给出手动 checklist（入口、操作、预期可见结果）。

## 硬性禁止项
- 防御性过度设计：无证据不加回退、重试、恢复路径。
- 范围蔓延：不修无关问题，不“顺手优化”。
- 虚假严谨：不写无意义测试，不测实现细节。
- 过早抽象：无当前需求不泛化。
- 静默行为变更：不得改变无关逻辑或引入隐性副作用。

## Philosophy
- Correct > Clever
- Simple > Flexible
- Verified > Assumed
- Minimal > Complete

## 不适用场景
以下场景不要使用本 skill：
- 头脑风暴
- 系统设计
- 规格文档编写
- UI 资产生成
- 纯内容创作
